# ADR-009: VRコントローラ操作マッピング改訂

## ステータス

**提案中（Proposed）**

## 日付

2026-03-23

## コンテキスト

1. **VRモード切替時にカメラ位置がリセットされる**: ADR-008の実装ではXRリググループ（`xrRigGroup`）の初期位置が原点(0,0,0)のままであり、デスクトップモードでルーム後端(z=3500)にいたユーザーがVRモードに入ると、Webページ埋め込み位置(z=0付近)に瞬間移動してしまう。VRセッション開始時にデスクトップカメラの位置をxrRigGroupに引き継ぐ必要がある。

2. **コントローラ入力が処理されない**: VRControlsManagerの`update()`が`requestAnimationFrame`ベースのフレームループから呼ばれているが、WebXRセッション中はブラウザのrAFコールバックが停止し、XRフレームコールバック（`setAnimationLoop`）のみが実行される。`update()`を`setAnimationLoop`コールバック内に移動する必要がある。

3. **操作マッピングの改善**: ADR-008のD38で定義した操作マッピング（左スティック=水平移動、右スティックY=上下移動）では、VR空間内での方向転換にヘッドトラッキングのみに依存しており、座位プレイやケーブル接続環境で不便である。ヨー回転をスティック入力で行えるようにし、より直感的な操作体系を提供する。

## 決定事項

### D40: VRセッション開始時のカメラ位置引き継ぎ

VRセッション開始時（`sessionstart`イベント）に、デスクトップモードのカメラ位置をxrRigGroupの位置に転写する。これにより、VRモードに入った際にデスクトップで見ていた位置からシームレスに体験を継続できる。

```typescript
webglRenderer.xr.addEventListener("sessionstart", () => {
  // カメラのワールド位置をxrRigGroupに転写
  xrRigGroup.position.copy(camera.position);
});
```

**根拠**: xrRigGroupはVRカメラの親であり、そのpositionがVR空間でのユーザーの基準位置となる。デスクトップカメラの位置を引き継ぐことで、ユーザーの空間的文脈が保たれる。

### D41: VRコントローラ入力のレンダリングループ統合

VRControlsManagerの`update()`呼び出しを、`requestAnimationFrame`ベースのフレームループから`setAnimationLoop`コールバック内に移動する。`setAnimationLoop`はWebXRセッション中もXRフレームコールバックとして確実に実行されるため、コントローラ入力が途切れることなく処理される。

```typescript
// scene.ts の setAnimationLoop 内
webglRenderer.setAnimationLoop(() => {
  // VRコントローラ入力処理（XRセッション中のみ）
  if (inVR) {
    onVRFrame(); // VRControlsManager.update() を呼ぶコールバック
  }
  // ... 既存のレンダリング処理
});
```

**根拠**: WebXR仕様では、XRフレームコールバック（`setAnimationLoop`経由で登録）のみがXRフレーム内で実行され、通常の`requestAnimationFrame`コールバックはXRセッション中は停止する。コントローラ入力はフレームごとにポーリングする必要があるため、レンダリングループ内に統合する必要がある。

### D42: VRコントローラ操作マッピング改訂

ADR-008のD38で定義した操作マッピングを以下のように改訂する。

**操作マッピング**:

| コントローラ     | 軸          | 操作                             |
| ---------------- | ----------- | -------------------------------- |
| 左サムスティック | Y軸（前後） | 前後併進移動（forward/backward） |
| 左サムスティック | X軸（左右） | 左右併進移動（strafe）           |
| 右サムスティック | Y軸（上下） | 上下移動（elevate/descend）      |
| 右サムスティック | X軸（左右） | ヨー回転（yaw rotation）         |

**ヨー回転の実装**:

- xrRigGroupのY軸回転（`rotation.y`）にオフセットを加算することで、ヘッドトラッキングに対するヨー回転オフセットを実現する。
- ヘッドトラッキング自体は変更せず、xrRigGroupの回転として適用されるため、ヘッドセットの物理的な向きとスティック入力による回転が自然に合成される。
- 回転速度はスムース回転（連続回転）とし、一定の角速度でフレームレート非依存（delta time乗算）とする。
- デッドゾーン（閾値: 0.15）を適用する。

**移動方向の基準**:

- 前後移動・左右strafe移動の方向は、xrRigGroupの回転（ヨーオフセット）とヘッドトラッキングの合成された向きを基準とする。
- 上下移動はワールド座標のY軸に沿って行い、向きに依存しない。

**根拠**: 左スティック=前後移動+左右strafe、右スティック=上下移動+ヨー回転というマッピングは、一般的なFPS/VRゲームのツインスティック操作に準拠する。左手で空間内の併進移動を統括し、右手で視点方向と高度を制御する分担により、両手の役割が明確で直感的に操作できる。ヨー回転をxrRigGroupの回転で実装することで、ヘッドトラッキングとの干渉を避けつつ自然な回転体験を提供する。

## スコープ外（将来のADR候補）

- **スナップターン**: 一定角度ごとの離散的な回転（酔い防止オプション）
- **移動速度調整UI**: VR内での移動速度変更
- **コンフォートオプション**: ビネット効果、テレポーテーション移動等の酔い対策
